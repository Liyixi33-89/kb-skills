# Tech-Debt-Tracker — 技术债务追踪器

## 目标

自动扫描项目历史版本中的 Code Review 报告和 Sprint 报告，汇总所有未修复的技术债务（🟡 建议修复项），生成结构化的债务看板，并在新版本开发前提供清理建议。

**核心隐喻**：你是一个"技术债务审计员"——定期盘点项目中的技术债务，确保没有问题被遗忘，防止债务累积到不可控的程度。

---

## 设计原则

1. **全量扫描**：扫描 `version-doc/` 下所有版本的 Review 报告和 Sprint 报告
2. **去重合并**：同一问题在多个版本中出现时，只保留最早的记录
3. **状态追踪**：区分"未修复"、"已修复"、"已忽略"三种状态
4. **优先级排序**：按严重程度和存在时间排序，越老的债务优先级越高
5. **可操作**：每条债务都附带修复建议和涉及文件

---

## 输入

| 参数 | 必填 | 说明 |
|------|------|------|
| **操作类型** | ✅ | `scan`（扫描生成报告）/ `check`（快速检查未修复数量）/ `resolve`（标记已修复） |
| **版本范围** | 否 | 指定扫描的版本范围（如 `v1.2.0~v1.3.0`），默认扫描全部 |
| **问题编号** | 否 | `resolve` 操作时指定要标记为已修复的问题编号 |

## 输出

| 文件 | 说明 |
|------|------|
| `version-doc/tech-debt-board.md` | 技术债务看板（持久化文件） |
| 控制台输出 | 债务摘要和建议 |

## 编排流程

### 模式 A：scan（全量扫描）

#### A-1：扫描版本目录

1. 读取 `version-doc/` 下所有 `v*` 目录，按版本号升序排列
2. 对每个版本目录，查找以下文件：
   - `review/` 目录下的所有 `.md` 文件（Code Review 报告）
   - `sprint-report.md`（Sprint 报告）

#### A-2：提取未修复问题

对每个 Code Review 报告：
1. 提取所有 🟡 建议修复的问题（编号、文件、问题描述、修复建议）
2. 提取所有 🔴 必须修复但标记为"下个版本修复"的问题

对每个 Sprint 报告：
1. 提取"技术债务"章节中的所有条目
2. 提取"建议下一步"中标记为"待修复"的条目

#### A-3：交叉验证修复状态

对每条提取到的债务：
1. 检查后续版本的 Review 报告中是否提到"已修复 CR-XXX"
2. 检查涉及的源码文件是否已被修改（通过读取文件检查）
3. 如果有明确证据表明已修复 → 标记为 ✅ 已修复
4. 如果无法确认 → 标记为 ⏳ 未修复

#### A-4：生成债务看板

```markdown
# 技术债务看板

> 最后扫描时间：{timestamp}
> 扫描版本范围：{v1.2.0 ~ v1.3.0}

## 📊 债务概览

| 指标 | 数量 |
|------|------|
| 总债务数 | {N} |
| ⏳ 未修复 | {N} |
| ✅ 已修复 | {N} |
| 🚫 已忽略 | {N} |
| 最老未修复债务 | {版本号} ({天数}天前) |

## ⏳ 未修复债务（按优先级排序）

### 🔴 高优先级（安全/正确性相关，或存在 > 2 个版本）

| # | 来源版本 | 编号 | 问题描述 | 涉及文件 | 修复建议 | 存在版本数 |
|---|---------|------|---------|---------|---------|-----------|
| 1 | v1.2.0 | CR-001 | IAgent 接口缺少 favoriteCount | Agent.ts | 补充接口字段 | 2 |

### 🟡 中优先级（代码质量/一致性相关）

| # | 来源版本 | 编号 | 问题描述 | 涉及文件 | 修复建议 | 存在版本数 |
|---|---------|------|---------|---------|---------|-----------|
| 1 | v1.3.0 | CR-002 | recalculateRatingStats 竞态条件 | reviewService.ts | 使用 MongoDB 事务 | 1 |

### 🟢 低优先级（优化建议）

| # | 来源版本 | 编号 | 问题描述 | 涉及文件 | 修复建议 | 存在版本数 |
|---|---------|------|---------|---------|---------|-----------|

## ✅ 已修复债务

| # | 来源版本 | 编号 | 问题描述 | 修复版本 |
|---|---------|------|---------|---------|
| 1 | v1.2.0 | CR-002 | FavoriteButton props 同步 | v1.2.0 (同版本修复) |
| 2 | v1.2.0 | CR-004 | API 层封装不一致 | v1.2.0 (同版本修复) |

## 📈 债务趋势

| 版本 | 新增债务 | 修复债务 | 净增 | 累计未修复 |
|------|---------|---------|------|-----------|
| v1.2.0 | 4 | 2 | +2 | 2 |
| v1.3.0 | 3 | 0 | +3 | 5 |
```

#### A-5：输出建议

根据债务数量和严重程度，输出清理建议：

| 未修复债务数 | 建议 |
|------------|------|
| 0 | ✅ 无技术债务，可以安心开发新功能 |
| 1-3 | 🟡 建议在下个版本的编码阶段顺带修复 |
| 4-6 | 🟠 建议安排 0.5-1d 专门清理技术债务 |
| > 6 | 🔴 建议先用 Refactor 工作流清理债务，再启动新 Feature |

---

### 模式 B：check（快速检查）

快速输出未修复债务数量，不生成完整报告：

1. 读取 `version-doc/tech-debt-board.md`（如存在）
2. 统计 ⏳ 未修复的数量
3. 输出：`📋 当前未修复技术债务：{N} 条（🔴 高优 {N} / 🟡 中优 {N} / 🟢 低优 {N}）`

如果看板文件不存在 → 提示先执行 `scan` 模式。

---

### 模式 C：resolve（标记已修复）

手动标记某条债务为已修复：

1. 读取 `version-doc/tech-debt-board.md`
2. 找到指定编号的债务条目
3. 将其从"⏳ 未修复"移到"✅ 已修复"，标注修复版本
4. 更新概览统计数字
5. 写回文件

## 与 Pipeline 的集成

`tech-debt-tracker` 在 Pipeline 中有两个触发点：

### 触发点 1：Pipeline 启动时（第 0.5 步）

`pipeline-orchestrator` 的第 0.5 步会调用 `tech-debt-tracker` 的 `check` 模式：
- 如果未修复债务 > 3 → 输出警告
- 如果未修复债务中有 🔴 高优先级 → 强制要求先修复

### 触发点 2：Pipeline 完成后

在 `changelog-gen` 之后，自动执行 `scan` 模式更新债务看板：
- 将本次 Code Review 中的 🟡 问题加入看板
- 检查本次是否修复了历史债务

---

## 优先级升级规则

债务的优先级会随时间自动升级：

| 条件 | 优先级变化 |
|------|-----------|
| 债务存在 1 个版本 | 保持原级别 |
| 债务存在 2 个版本 | 🟢 → 🟡 |
| 债务存在 3+ 个版本 | 🟡 → 🔴 |
| 安全相关的债务 | 始终为 🔴 |

---

## 约束

1. **不自动修复**：本 Skill 只追踪和提醒，不直接修改代码
2. **看板文件是唯一真相源**：所有债务状态以 `tech-debt-board.md` 为准
3. **不重复计数**：同一问题在多个版本中出现时，只计为一条债务
4. **已忽略的债务不再提醒**：标记为 🚫 已忽略的债务不计入未修复统计

---

## 边界条件处理

| 场景 | 处理方式 |
|------|---------|
| `version-doc/` 目录不存在 | 提示"无版本历史，无法扫描技术债务" |
| 某个版本没有 Review 报告 | 跳过该版本，标注"⚠️ {版本号} 无 Review 报告" |
| 看板文件不存在（首次运行） | 创建新的看板文件 |
| 债务涉及的文件已被删除 | 标记为 ✅ 已修复（文件不存在 = 问题不存在） |
