# Refactor — 代码重构

## 目标

对指定的代码模块进行结构化重构，改善代码质量（可读性、可维护性、可测试性），同时确保不破坏现有功能。

**核心隐喻**：你是一个"代码整形师"——改善代码的"外观"（结构），不改变它的"行为"（功能）。每一步重构都小而安全。

---

## 设计原则

1. **行为不变**：重构前后功能完全一致，不趁机加功能或改逻辑
2. **小步前进**：每次只做一种重构操作，不同时做多种
3. **KB 对齐**：重构后的代码应更符合 KB 中记录的项目规范
4. **可验证**：每步重构后都可以运行测试验证
5. **有理有据**：每个重构决策都说明原因（解决什么 Code Smell）

---

## 支持的重构模式

| 重构模式 | 适用场景 | 说明 |
|---------|---------|------|
| **提取函数** | 函数过长（>50 行）、重复代码块 | 将代码块提取为独立函数 |
| **提取组件** | React 组件过大（>200 行）、JSX 嵌套过深 | 将 UI 片段提取为子组件 |
| **提取 Hook** | 多个组件有相同的状态逻辑 | 提取为自定义 Hook |
| **状态提升/下沉** | 状态放置层级不合理 | 调整状态的所有权 |
| **依赖解耦** | 模块之间耦合过紧 | 引入接口/中间层解耦 |
| **简化条件** | 复杂的 if-else 嵌套 | 使用早期返回、策略模式 |
| **合并重复** | 多处相似代码 | 提取公共函数/组件 |
| **重命名** | 命名不清晰或不符合规范 | 统一命名风格 |
| **文件拆分** | 单文件过大（>300 行） | 按职责拆分为多个文件 |

---

## 输入

| 参数 | 必填 | 说明 |
|------|------|------|
| **目标文件/模块** | ✅ | 需要重构的文件路径或模块名称 |
| **重构目标** | 否 | 指定重构方向（如"拆分组件""提取函数"），默认自动分析 |
| **约束** | 否 | 重构的限制条件（如"不改变文件结构""不拆分文件"） |

## 输出

| 输出 | 说明 |
|------|------|
| 重构方案 | Code Smell 分析 + 重构计划 |
| 重构代码 | 直接修改源码文件 |
| 变更摘要 | 修改了哪些文件、做了什么重构 |

---

## 编排流程

### 第 1 步：分析代码异味

读取目标文件，识别 Code Smell：

| Code Smell | 检测标准 | 重构方式 |
|-----------|---------|---------|
| 过长函数 | 函数 > 50 行 | 提取函数 |
| 过大组件 | React 组件 > 200 行 | 提取子组件 |
| 过大文件 | 文件 > 300 行 | 文件拆分 |
| 重复代码 | 相似代码块出现 2+ 次 | 提取公共函数 |
| 深层嵌套 | if/for 嵌套 > 3 层 | 早期返回、提取函数 |
| 过长参数列表 | 函数参数 > 4 个 | 参数对象化 |
| 魔法数字/字符串 | 硬编码的常量 | 提取为命名常量 |
| 不一致命名 | 与 KB 中的命名规范不符 | 重命名 |
| 状态管理混乱 | 全局状态过多、props drilling | 状态提升/下沉、Context |
| 未使用代码 | 未引用的变量/函数/导入 | 删除 |

### 第 2 步：查阅 KB 确定目标模式

读取 KB 中同类型文件的模式，确定重构后应该达到的标准：
- 文件组织方式
- 函数/组件的合理大小
- 命名规范
- 导入组织方式

### 第 3 步：制定重构计划

```markdown
### 重构计划

**目标文件**：{filePath}
**当前问题**：
1. {Code Smell 1}：{具体描述}
2. {Code Smell 2}：{具体描述}

**重构步骤**（按顺序执行）：

| 步骤 | 重构操作 | 影响范围 | 风险 |
|------|---------|---------|------|
| 1 | 提取 {functionName} 函数 | 当前文件 | 低 |
| 2 | 提取 {ComponentName} 组件 | 当前文件 + 新文件 | 中 |
| 3 | 重命名 {oldName} → {newName} | 多个文件 | 中 |
```

### 第 4 步：逐步执行重构

**每步重构都遵循**：
1. 明确说明这步做什么、为什么做
2. 执行代码修改
3. 列出需要验证的点

**重构代码规范**：
- 保持与周围代码风格一致
- 新提取的函数/组件添加 TypeScript 类型
- 新文件遵循项目的文件组织规范
- 更新所有引用（导入路径、函数调用等）

### 第 5 步：验证清单

```markdown
### 重构验证清单

- [ ] 所有修改的文件无 TypeScript 编译错误
- [ ] 所有导入路径正确
- [ ] 功能行为与重构前一致
- [ ] 新提取的函数/组件有完整的类型定义
- [ ] 命名符合项目规范
```

### 第 6 步：输出摘要

```
## 重构完成

**目标**：{重构目标}
**解决的 Code Smell**：{N} 个

| # | 重构操作 | 文件 | 说明 |
|---|---------|------|------|
| 1 | {操作} | {file} | {说明} |

建议下一步：
1. 运行项目确认功能正常
2. 使用 `code-review` 审查重构后的代码
3. 使用 `gen-test-code` 补充测试（如缺失）
4. 执行 `doc-code-to-kb` 更新知识库
```

---

## 约束

### 重构约束

1. **不改变行为**：重构前后功能完全一致
2. **不加新功能**：重构过程中不添加任何新功能
3. **不删除功能**：不以"简化"为由删除现有功能
4. **保持向后兼容**：如果重构涉及公共 API/组件 Props，保持接口不变
5. **一次一种**：每步只做一种重构操作

### 边界条件处理

| 场景 | 处理方式 |
|------|---------|
| 文件无明显 Code Smell | 告知用户代码质量良好，无需重构 |
| 重构影响范围过大 | 分阶段执行，先处理高优先级的 Code Smell |
| 重构需要修改公共接口 | 标注 `⚠️ 破坏性变更`，建议先评估影响 |
| KB 不存在 | 按通用最佳实践重构，标注"⚠️ 未参考 KB" |
| 无测试覆盖 | 建议先补充测试再重构 |
