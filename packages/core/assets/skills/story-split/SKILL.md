# Story-Split — 用户故事拆分

## 目标

将 PRD 中的功能需求拆分为可执行的用户故事，按 Epic → Story → Task 三级结构组织，每个 Story 包含完整的上下文、实现细节、验收标准和依赖关系，让开发人员可以独立领取并完成。

**核心隐喻**：你是一个"Scrum Master"——把大象切成可以吃的小块。每块都能独立交付价值，每块都有清晰的完成标准。

---

## 设计原则

1. **独立可交付**：每个 Story 都能独立开发、独立测试、独立上线（或至少独立 Demo）
2. **上下文完整**：每个 Story 包含足够的背景信息，开发人员不需要读完整个 PRD 就能开始
3. **依赖显式化**：Story 之间的依赖关系明确标注，避免并行开发时的冲突
4. **估算可行**：每个 Story 的工作量在 1-3 天内，超过 3 天的需要继续拆分
5. **优先级排序**：按业务价值 + 技术依赖综合排序，确定实施顺序

---

## 输入

| 参数 | 必填 | 说明 |
|------|------|------|
| **PRD 文件** | ✅ | `version-doc/{版本号}/prd/prd.md` |
| **版本号** | ✅ | 如 `v1.0.1`，用于确定输入/输出目录 |
| **Sprint 周期** | 否 | 默认 2 周，用于估算每个 Sprint 可完成的 Story 数 |
| **团队规模** | 否 | 默认 1 人（AI 辅助开发），影响并行度建议 |

**版本号获取规则**：
1. 如果用户直接指定了版本号 → 使用用户指定的
2. 如果文件路径中包含版本号 → 从路径提取
3. 如果以上未提供 → **向用户询问版本号**，不可跳过

## 输出

**输出位置**：`version-doc/{版本号}/stories/`

| 文件 | 说明 |
|------|------|
| `epic-overview.md` | Epic 总览（所有 Epic 和 Story 的索引 + 依赖图） |
| `story-{编号}-{slug}.md` | 每个 Story 一个文件 |

## 编排流程

### 第 0 步：检查前置条件

1. 检查 `version-doc/{版本号}/prd/prd.md` 是否存在
   - 不存在 → 提示用户先执行 `prd-brd-to-prd` Skill
2. 检查 `version-doc/{版本号}/stories/` 是否已有文件
   - 已有 → 读取现有 Story，询问用户是"覆盖重写"还是"增量添加"
3. 检查是否有设计文档（`version-doc/{版本号}/design/`）
   - 有 → 读取，用于更精确的工作量估算和技术依赖分析

### 第 1 步：读取 PRD 并提取功能模块

读取 `version-doc/{版本号}/prd/prd.md`，提取：
- 所有功能模块及其优先级
- 每个功能模块的验收标准
- 接口约定和数据模型约定
- 非功能需求

### 第 2 步：查阅 KB 分析技术依赖

读取 KB 中的现有架构信息：
- 现有 Model 列表 → 判断哪些 Story 需要新建 Model
- 现有 API 列表 → 判断哪些 Story 需要新建/修改 API
- 现有页面列表 → 判断哪些 Story 需要新建/修改页面
- 现有组件列表 → 判断哪些 Story 可以复用组件

### 第 3 步：Epic 拆分

将 PRD 中的功能模块映射为 Epic：

```markdown
## Epic 1: {功能模块名}
> 来源：PRD 功能模块 {N}
> 优先级：{P0/P1/P2}

**目标**：{一句话描述 Epic 的业务目标}
**涉及层级**：{后端 Model / 后端 API / 前端页面 / 前端组件}
```

**Epic 拆分规则**：
- 一个 PRD 功能模块 = 一个 Epic（通常）
- 如果功能模块过大（涉及 > 5 个页面或 > 10 个 API），拆分为多个 Epic
- 跨模块的公共基础设施（如新增公共组件、新增中间件）单独成 Epic

### 第 4 步：Story 拆分

对每个 Epic，按以下维度拆分为 Story：

**纵向拆分（按功能切片）**：
- 每个独立的用户操作流程 = 一个 Story
- 例如："创建 Agent" 和 "编辑 Agent" 是两个 Story

**横向拆分（按技术层级）**：
- 如果一个功能涉及多个层级且工作量大，可以按层级拆分
- 例如："Agent 标签 - 后端 API" 和 "Agent 标签 - 前端 UI" 是两个 Story

**Story 编号规则**：`{Epic编号}.{Story序号}`，如 `1.1`、`1.2`、`2.1`

**Story 粒度标准**：
| 工作量 | 处理方式 |
|--------|---------|
| < 0.5 天 | 合并到相关 Story 中 |
| 0.5 - 3 天 | ✅ 合适的 Story 粒度 |
| 3 - 5 天 | 考虑继续拆分 |
| > 5 天 | 必须继续拆分 |

### 第 5 步：分析依赖关系

对所有 Story，分析依赖关系：

**技术依赖**：
- Model Story 必须先于使用该 Model 的 Service/Route Story
- 后端 API Story 必须先于调用该 API 的前端 Story
- 公共组件 Story 必须先于使用该组件的页面 Story

**业务依赖**：
- 基础功能 Story 先于增强功能 Story
- 列表页 Story 先于详情页 Story（通常）

**依赖图格式**：
```
Story 1.1 (Model) → Story 1.2 (API) → Story 1.3 (前端页面)
                                     ↗
Story 2.1 (公共组件) ─────────────────
```

### 第 6 步：排定优先级和实施顺序

综合考虑：
1. **业务优先级**（PRD 中的 P0 > P1 > P2）
2. **技术依赖**（被依赖的 Story 排在前面）
3. **风险**（高风险 Story 排在前面，尽早暴露问题）

输出实施顺序：
```markdown
### 建议实施顺序

| 顺序 | Story | 估算工时 | 依赖 | 说明 |
|------|-------|---------|------|------|
| 1 | 1.1 Agent Model 扩展 | 0.5d | 无 | 基础数据层 |
| 2 | 1.2 标签 API | 1d | 1.1 | 后端接口 |
| 3 | 2.1 TagSelector 组件 | 1d | 无 | 可并行 |
| 4 | 1.3 标签筛选 UI | 1.5d | 1.2, 2.1 | 前端页面 |
```

### 第 7 步：生成 Story 文件

对每个 Story，生成独立的 Story 文件：

```markdown
# Story {编号}: {标题}

> Epic: {Epic 名称}
> 来源: PRD 功能模块 {N}
> 优先级: {P0/P1/P2}
> 估算工时: {N} 天
> 依赖: {依赖的 Story 编号列表，或"无"}

## 背景

{为什么要做这个 Story，业务上下文}

## 目标

{这个 Story 完成后，用户/系统能做什么}

## 实现范围

### 需要新增的文件
| 文件 | 类型 | 说明 |
|------|------|------|
| {filePath} | Model/Route/Service/Page/Component | {说明} |

### 需要修改的文件
| 文件 | 修改内容 | 说明 |
|------|---------|------|
| {filePath} | {具体修改} | {说明} |

## 技术细节

{关键的技术实现要点，如数据库 Schema 变更、API 接口定义、组件 Props 等}

## 验收标准

- [ ] {验收条件 1}
- [ ] {验收条件 2}
- [ ] {验收条件 3}

## 测试要点

- {需要测试的关键场景}

## 注意事项

- {开发时需要注意的坑或风险}
```

### 第 8 步：生成 Epic 总览

```markdown
# Epic 总览 — {版本号}

## 版本目标

{一句话描述本版本的核心目标}

## Epic 列表

| Epic | 名称 | Story 数 | 估算总工时 | 优先级 |
|------|------|---------|-----------|--------|
| E1 | {名称} | {N} | {N}d | P0 |
| E2 | {名称} | {N} | {N}d | P1 |

## 依赖关系图

{ASCII 或 Mermaid 格式的依赖图}

## 建议 Sprint 规划

### Sprint 1（第 1-2 周）
| Story | 标题 | 工时 |
|-------|------|------|
| 1.1 | ... | 0.5d |
| 1.2 | ... | 1d |

### Sprint 2（第 3-4 周）
| Story | 标题 | 工时 |
|-------|------|------|
| 1.3 | ... | 1.5d |
```

### 第 9 步：自检

- [ ] 每个 PRD 功能模块都有对应的 Epic
- [ ] 每个 Epic 都有至少一个 Story
- [ ] 每个 Story 的工作量在 0.5-3 天内
- [ ] 依赖关系无循环
- [ ] 实施顺序尊重依赖关系
- [ ] 每个 Story 都有验收标准

### 第 10 步：输出摘要

```
## 用户故事拆分完成

- PRD 来源：version-doc/{版本号}/prd/prd.md
- Epic 数：{N} 个
- Story 数：{N} 个
- 估算总工时：{N} 天

输出文件：
- version-doc/{版本号}/stories/epic-overview.md
- version-doc/{版本号}/stories/story-{编号}-{slug}.md × {N}

建议下一步：
1. 按实施顺序开始第一个 Story
2. 使用 `prd-to-backend-design` / `prd-to-frontend-design` 为每个 Story 生成技术设计
3. 每完成一个 Story，使用 `code-review` 进行代码审查
```

---

## 约束

### 拆分约束

1. **不超出 PRD 范围**：Story 中的功能必须可追溯到 PRD
2. **不遗漏 PRD 需求**：每个 PRD 功能模块都必须被至少一个 Story 覆盖
3. **Story 独立性**：每个 Story 完成后系统仍可正常运行（不引入半成品）
4. **验收标准可测试**：每个验收条件都是可验证的（不是"用户体验好"这种模糊描述）

### 边界条件处理

| 场景 | 处理方式 |
|------|---------|
| PRD 不存在 | 提示用户先生成 PRD |
| PRD 需求过于模糊 | 在 Story 中标注 `⚠️ 待细化`，建议先完善 PRD |
| 功能模块之间高度耦合 | 合并为一个 Epic，在 Story 层面拆分 |
| 单个功能过于简单 | 合并到相关 Story 中，不单独成 Story |
| KB 不存在 | 标注"⚠️ 未参考 KB"，按通用模式估算 |
