# PRD-BRD-TO-PRD — BRD 转产品需求文档

## 目标

将标准化 BRD（`brd_normalized.md`）转化为可直接指导技术设计和开发的 PRD 文档。PRD 在 BRD 基础上补充：交互细节、信息架构、页面流转、接口约定、非功能需求、验收标准。

**核心隐喻**：你是一个"产品经理"——把业务方的"要什么"翻译成开发团队的"做什么"。BRD 说的是目标，PRD 说的是方案。

---

## 设计原则

1. **BRD 为纲**：PRD 中的每个功能点都必须可追溯到 BRD 中的某个需求，不凭空新增功能
2. **KB 为据**：设计方案必须参考项目知识库中的现有架构，复用已有组件/API/Model，避免重复建设
3. **开发可执行**：PRD 的粒度必须足够细，开发人员读完 PRD 就能开始技术设计，不需要再猜
4. **验收可测**：每个功能点都有明确的验收标准，QA 可以直接据此编写测试用例
5. **显式标注新增 vs 复用**：明确标注哪些是新增的、哪些是复用现有的、哪些是修改现有的

---

## 输入

| 参数 | 必填 | 说明 |
|------|------|------|
| **BRD 文件** | ✅ | `version-doc/{版本号}/brd/brd_normalized.md`（必须是最终版，无 TBD 占位符） |
| **版本号** | ✅ | 如 `v1.0.1`，用于确定输入/输出目录 |
| **目标应用** | 否 | `web` / `admin` / `both`，默认根据 BRD 内容自动判断 |

**版本号获取规则**：
1. 如果用户直接指定了版本号 → 使用用户指定的
2. 如果 BRD 文件路径中包含版本号 → 从路径提取
3. 如果以上未提供 → **向用户询问版本号**，不可跳过

## 输出

**输出位置**：`version-doc/{版本号}/prd/`

| 文件 | 说明 |
|------|------|
| `prd.md` | 完整的产品需求文档 |

## 编排流程

### 第 0 步：检查前置条件

1. 检查 `version-doc/{版本号}/brd/brd_normalized.md` 是否存在
   - 不存在 → 提示用户先执行 `brd-normalize` Skill
2. 检查 BRD 文档状态是否为"最终版"
   - 如果仍有 `[TBD-xxx]` 占位符 → 提示用户先完成 BRD 澄清流程
3. 检查 `version-doc/{版本号}/prd/prd.md` 是否已存在
   - 已存在 → 读取内容，询问用户是"覆盖重写"还是"增量更新"

### 第 1 步：读取 BRD 并提取需求清单

读取 `version-doc/{版本号}/brd/brd_normalized.md`，提取：
- 背景与目标
- 需求清单（编号、名称、优先级、描述）
- 每个需求的详情（用户故事、角色、验收标准、原始表述）
- 术语表

### 第 2 步：查阅项目知识库

根据 BRD 中涉及的功能模块，从 KB 中查找现有实现：

1. 读取 `kb/server/server/00_project_map.md` 了解项目整体架构
2. 根据需求涉及的模块，读取对应索引：
   - 涉及数据存储 → `kb/server/server/02_index_model.md`
   - 涉及 API 接口 → `kb/server/server/01_index_api.md`
   - 涉及业务逻辑 → `kb/server/server/03_index_service.md`
   - 涉及前端页面 → `kb/frontend/@agency/{app}/01_index_page.md`
   - 涉及前端组件 → `kb/frontend/@agency/{app}/02_index_component.md`
   - 涉及前端 API → `kb/frontend/@agency/{app}/03_index_api.md`
   - 涉及状态管理 → `kb/frontend/@agency/{app}/04_index_store.md`
   - 涉及类型定义 → `kb/frontend/@agency/{app}/05_index_types.md`

3. 整理出：
   - **可复用的现有资源**（已有的 API、组件、Model、Service）
   - **需要修改的现有资源**（需要扩展字段、新增端点等）
   - **需要新建的资源**（全新的 Model、页面、Service 等）

### 第 3 步：逐需求细化 PRD

对 BRD 中的每个需求，逐一细化为 PRD 级别的功能描述：

#### 3.1 功能描述细化

- 将 BRD 的"做什么"转化为"怎么做"
- 补充交互细节：用户操作步骤、页面流转、异常处理
- 补充信息架构：页面布局、数据展示方式、表单字段

#### 3.2 接口约定

对每个需要后端支持的功能，定义接口约定：

```markdown
| 接口 | 方法 | 路径 | 说明 | 状态 |
|------|------|------|------|------|
| 获取列表 | GET | /api/xxx | 分页获取 | 新增 |
| 创建记录 | POST | /api/xxx | 创建新记录 | 新增 |
| 更新状态 | PUT | /api/xxx/:id | 更新记录 | 复用现有 |
```

**状态**标注：`新增` / `复用现有` / `修改现有`（附说明）

#### 3.3 数据模型约定

对需要新增/修改的数据模型，定义字段约定：

```markdown
| 字段 | 类型 | 必填 | 说明 | 状态 |
|------|------|------|------|------|
| name | string | ✅ | 名称 | 新增 |
| status | string | ✅ | 状态枚举 | 新增 |
| userId | ObjectId | ✅ | 关联用户 | 复用 |
```

#### 3.4 页面设计

对每个涉及的前端页面，描述：
- 页面入口和路由
- 页面布局（文字描述，不画图）
- 核心交互流程（步骤列表）
- 使用的组件（标注复用/新建）

#### 3.5 验收标准

将 BRD 中的初步验收标准细化为可测试的条件：
- 正向场景（Happy Path）
- 异常场景（Error Cases）
- 边界条件（Edge Cases）
- 权限场景（不同角色的行为差异）

### 第 4 步：补充非功能需求

根据需求特点，补充以下非功能需求（按需）：

| 维度 | 检查项 |
|------|--------|
| 性能 | 列表页加载时间、搜索响应时间、并发量 |
| 安全 | 认证要求、权限控制、数据脱敏、输入校验 |
| 兼容性 | 浏览器兼容、移动端适配 |
| 可用性 | 加载状态、空状态、错误提示 |
| 数据 | 数据迁移、历史数据兼容 |

### 第 5 步：生成 PRD 文档

读取 `{SKILL_DIR}/references/prd-template.md`，按模板生成 `prd.md`。

**关键要求**：
- 每个功能点都标注 BRD 溯源（`来源：BRD 需求 {N}`）
- 每个接口/模型/页面都标注状态（新增/复用/修改）
- 验收标准必须可测试，不能是模糊描述

### 第 5.5 步：PRD 质量门禁（🆕 强制步骤）

> 确保 PRD 质量稳定，防止出现 v1.2.0 那样过于简略的 PRD。
> 此步骤**不可跳过**，PRD 必须通过所有必选检查项才能输出。

**检查清单**：

| # | 检查项 | 级别 | 说明 |
|---|--------|------|------|
| 1 | **交互流程** | 🔴 必须 | 每个功能模块必须包含主流程（步骤列表）和至少 1 个异常流程 |
| 2 | **异常场景** | 🔴 必须 | 每个功能模块必须列出异常场景的处理方式（未登录、参数错误、网络失败等） |
| 3 | **验收标准** | 🔴 必须 | 每个功能模块必须包含可测试的验收标准（checkbox 格式），至少覆盖：正向场景、异常场景 |
| 4 | **接口约定** | 🔴 必须 | 涉及后端的功能必须有接口约定表格（方法、路径、说明、状态） |
| 5 | **数据模型约定** | 🔴 必须 | 涉及数据存储的功能必须有数据模型约定表格（字段、类型、必填、说明、状态） |
| 6 | **非功能需求** | 🔴 必须 | PRD 必须包含非功能需求章节，至少覆盖：性能、安全、可用性 |
| 7 | **页面设计** | 🟡 推荐 | 涉及前端的功能应有页面设计描述（路由、布局、组件） |
| 8 | **BRD 溯源** | 🔴 必须 | 每个功能模块必须标注 `来源：BRD 需求 {N}` |
| 9 | **KB 参考** | 🟡 推荐 | PRD 应包含 KB 参考章节，列出参考的 KB 文件 |
| 10 | **开放问题** | 🟡 推荐 | 如有未确定的决策点，应列在开放问题章节 |

**执行方式**：

1. 生成 PRD 后，逐项检查上述清单
2. 对每个 🔴 必须项：
   - 通过 → ✅
   - 未通过 → 自动补充缺失内容，重新写入 `prd.md`
3. 对每个 🟡 推荐项：
   - 未通过 → 在 PRD 末尾标注 `⚠️ 建议补充：{检查项名称}`
4. 所有 🔴 必须项通过后，才能进入第 6 步

**质量评分**：

| 评分 | 条件 |
|------|------|
| ⭐⭐⭐⭐⭐ 优秀 | 10/10 项全部通过 |
| ⭐⭐⭐⭐ 良好 | 所有 🔴 通过 + 2/3 🟡 通过 |
| ⭐⭐⭐ 合格 | 所有 🔴 通过 |
| ⭐⭐ 不合格 | 有 🔴 未通过 → 必须补充后重新检查 |

## 约束

### 内容约束

1. **不超出 BRD 范围**：PRD 中的每个功能都必须能追溯到 BRD 中的需求，不得自行添加 BRD 未提及的功能
2. **不做技术决策**：PRD 只定义"做什么"和"做到什么程度"，不决定"用什么技术实现"（那是技术设计文档的事）
3. **接口约定是约定不是设计**：PRD 中的接口路径、字段是建议性的，技术设计阶段可以调整
4. **复用优先**：如果 KB 中已有可复用的组件/API/Model，优先标注复用，不要重新设计

### 格式约束

1. **表格优先**：接口列表、字段列表、验收标准等用表格展示
2. **BRD 溯源必标**：每个功能模块必须标注 `来源：BRD 需求 {N}`
3. **状态必标**：每个接口/模型/页面必须标注 `新增` / `复用现有` / `修改现有`

### 边界条件处理

| 场景 | 处理方式 |
|------|---------|
| BRD 仍有 TBD 占位符 | 拒绝生成，提示先完成 BRD 澄清 |
| KB 目录不存在 | 提示用户先执行 `doc-code-to-kb` 生成知识库，但仍可生成 PRD（标注"⚠️ 未参考 KB"） |
| BRD 需求过于模糊 | 在 PRD 中标注 `⚠️ 待细化` 并给出细化建议 |
| 需求与现有功能冲突 | 在 PRD 中标注冲突点，建议在技术设计阶段解决 |

## Few-Shot 示例

### 示例：BRD 需求 → PRD 功能模块

**BRD 需求**：
> 需求 1：Agent 标签功能
> 描述：允许用户给 Agent 打标签，支持按标签筛选 Agent 列表。
> 优先级：P1

**PRD 功能模块**：

```markdown
### 功能模块 1：Agent 标签管理

> 来源：BRD 需求 1

#### 1.1 功能描述

用户可以为自己创建的 Agent 添加/删除标签（Tag），并在 Agent 列表页通过标签筛选。

#### 1.2 交互流程

**添加标签**：
1. 用户进入 Agent 详情页 → 点击"添加标签"按钮
2. 弹出标签选择器（支持选择已有标签 + 输入新标签）
3. 选择/输入后点击确认 → 标签显示在 Agent 卡片上
4. 标签数量上限：10 个/Agent

**按标签筛选**：
1. 用户在 Agent 列表页顶部看到标签筛选栏
2. 点击标签 → 列表只显示包含该标签的 Agent
3. 支持多标签 AND 筛选
4. 点击"清除筛选" → 恢复全量列表

#### 1.3 接口约定

| 接口 | 方法 | 路径 | 说明 | 状态 |
|------|------|------|------|------|
| 获取 Agent 列表 | GET | /api/agents | 增加 tags 查询参数 | 修改现有 |
| 更新 Agent 标签 | PUT | /api/agents/:id | body 中增加 tags 字段 | 修改现有 |
| 获取标签列表 | GET | /api/tags | 获取当前用户的所有标签 | 新增 |

#### 1.4 数据模型约定

**Agent Model（修改现有）**：

| 字段 | 类型 | 必填 | 说明 | 状态 |
|------|------|------|------|------|
| tags | string[] | 否 | 标签数组，最多 10 个 | 新增字段 |

#### 1.5 页面设计

**AgentsPage（修改现有）**：
- 列表顶部新增标签筛选栏（水平排列的 Tag 组件）
- Agent 卡片上显示标签（Antd Tag 组件，复用现有）

**AgentDetailPage（修改现有）**：
- 详情区域新增"标签"模块
- 包含已有标签展示 + "添加标签"按钮

#### 1.6 验收标准

**正向场景**：
- [ ] 用户可以为 Agent 添加 1-10 个标签
- [ ] 用户可以删除已有标签
- [ ] Agent 列表页可以按单个标签筛选
- [ ] Agent 列表页可以按多个标签 AND 筛选
- [ ] 清除筛选后恢复全量列表

**异常场景**：
- [ ] 添加超过 10 个标签时提示"标签数量已达上限"
- [ ] 标签名称为空时不允许添加
- [ ] 标签名称超过 20 字符时提示"标签名称过长"

**权限场景**：
- [ ] 用户只能管理自己创建的 Agent 的标签
- [ ] 其他用户查看 Agent 时可以看到标签但不能编辑
```
