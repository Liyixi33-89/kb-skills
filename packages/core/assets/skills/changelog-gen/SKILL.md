# Changelog-Gen — 变更日志生成

## 目标

基于版本开发过程中产生的所有文档（PRD、设计文档、Story）和代码变更，自动生成结构化的变更日志，用于版本发布时的沟通和记录。

**核心隐喻**：你是一个"技术写作者"——把开发团队做的事翻译成用户和管理层能理解的语言。既要准确，又要易读。

---

## 设计原则

1. **双视角**：同时生成面向用户的变更说明（功能维度）和面向开发者的技术变更（文件维度）
2. **PRD 溯源**：每个变更都关联到 PRD 需求编号，确保可追溯
3. **语义化分类**：按 Conventional Commits 规范分类（feat/fix/breaking/perf/refactor）
4. **自动化**：尽可能从已有文档中提取信息，减少手动编写

---

## 输入

| 参数 | 必填 | 说明 |
|------|------|------|
| **版本号** | ✅ | 如 `v1.0.1`，用于定位版本文档和命名 |
| **上一版本号** | 否 | 如 `v1.0.0`，用于对比变更范围。默认为首个版本 |
| **发布日期** | 否 | 默认为当天日期 |

**版本号获取规则**：
1. 如果用户直接指定了版本号 → 使用用户指定的
2. 如果以上未提供 → **向用户询问版本号**，不可跳过

## 输出

**输出位置**：`version-doc/{版本号}/`

| 文件 | 说明 |
|------|------|
| `CHANGELOG.md` | 完整的变更日志 |

同时更新项目根目录的 `CHANGELOG.md`（追加本版本内容到顶部）。

---

## 编排流程

### 第 0 步：检查前置条件

1. 检查 `version-doc/{版本号}/` 目录是否存在
2. 检查是否有 PRD（`version-doc/{版本号}/prd/prd.md`）
3. 检查是否有设计文档（`version-doc/{版本号}/design/`）
4. 检查是否有 Story 文件（`version-doc/{版本号}/stories/`）
5. **检查 CHANGELOG 是否已存在**（🆕 新增）：
   - 如果 `version-doc/{版本号}/CHANGELOG.md` 已存在 → 读取内容
     - 如果内容与当前需求**一致**（同一功能的更新）→ **覆盖模式**（重新生成）
     - 如果内容与当前需求**不同**（不同功能的 CHANGELOG）→ ⛔ **拒绝覆盖**，输出：
       ```
       ❌ CHANGELOG 冲突：version-doc/{版本号}/CHANGELOG.md 已包含「{已有功能名}」的变更日志。
       当前需求与已有 CHANGELOG 不匹配。
       
       建议：
       1. 使用新的版本号
       2. 或手动合并两个功能的 CHANGELOG
       ```
   - 如果不存在 → 正常创建

### 第 1 步：收集变更信息

从以下来源收集变更信息：

#### 1.1 从 PRD 提取功能变更

读取 `version-doc/{版本号}/prd/prd.md`，提取：
- 所有功能模块名称和描述
- 每个功能的优先级
- 验收标准（用于描述功能效果）

#### 1.2 从设计文档提取技术变更

读取 `version-doc/{版本号}/design/` 下的所有文件，提取：
- 新增/修改的 API 端点
- 新增/修改的数据模型
- 新增/修改的页面
- 新增/修改的组件

#### 1.3 从 Story 提取实施详情

读取 `version-doc/{版本号}/stories/` 下的所有文件，提取：
- 每个 Story 的标题和描述
- 实现范围（新增/修改的文件列表）

### 第 2 步：分类变更

将收集到的变更按以下类别分类：

| 类别 | 标识 | 说明 | 来源 |
|------|------|------|------|
| ✨ 新功能 | `feat` | 新增的用户可感知功能 | PRD 中的新增功能 |
| 🐛 Bug 修复 | `fix` | 修复的已知问题 | Bug 修复记录 |
| 💥 破坏性变更 | `breaking` | 不兼容的 API/数据变更 | 设计文档中的 Schema 变更 |
| ⚡ 性能优化 | `perf` | 性能相关的改进 | 设计文档中的性能优化 |
| ♻️ 重构 | `refactor` | 代码重构（不影响功能） | Story 中的重构任务 |
| 📝 文档 | `docs` | 文档更新 | — |
| 🏗️ 基础设施 | `infra` | 构建/部署/配置变更 | — |


### 第 4 步：自检

- [ ] 每个 PRD 功能都在 Changelog 中有对应条目
- [ ] 破坏性变更都有迁移指南
- [ ] 版本概要对非技术人员可读
- [ ] 技术变更明细完整（所有新增/修改的文件都列出）

### 第 5 步：输出摘要

```
## 变更日志生成完成

- 版本：{版本号}
- 发布日期：{date}
- 输出文件：version-doc/{版本号}/CHANGELOG.md

变更统计：
- ✨ 新功能：{N} 个
- 🐛 Bug 修复：{N} 个
- 💥 破坏性变更：{N} 个

建议下一步：
1. 审阅变更日志，确认内容准确
2. 将变更日志同步到项目根目录 CHANGELOG.md
3. 发布版本
```

---

## 约束

### 内容约束

1. **不编造变更**：所有变更必须有文档来源（PRD/设计文档/Story）
2. **用户视角优先**：新功能描述用用户能理解的语言，不用技术术语
3. **破坏性变更必须醒目**：使用 ⚠️ 标记，并提供迁移指南
4. **版本概要必须简洁**：不超过 3 句话

### 边界条件处理

| 场景 | 处理方式 |
|------|---------|
| PRD 不存在 | 从设计文档和 Story 中提取变更信息 |
| 设计文档不存在 | 从 PRD 和 Story 中提取 |
| Story 不存在 | 从 PRD 和设计文档中提取 |
| 所有文档都不存在 | 提示用户提供变更信息 |
| 无破坏性变更 | 省略该章节 |
| 无 Bug 修复 | 省略该章节 |
