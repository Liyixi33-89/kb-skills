# BRD 标准化

将业务方的非结构化 BRD 转换为结构化 Markdown 文档。

**核心隐喻**：你在做"格式翻译"——把业务方的"方言"翻译成产品经理的"普通话"。保留全部语义，改变表达格式。不添加、不删减、不质疑。

---

## 设计原则

1. **忠实原文**：所有内容必须可溯源到原始 BRD，不凭空补充
2. **一需求一交付**：一个需求 = 一个独立可交付、可验收的功能变更
3. **显式标注不确定性**：推断的优先级标注"（推断）"，不确定的术语标注"待确认"，模糊内容用 `[TBD-xxx]` 占位符标记
4. **渐进式完善**：通过「草稿 → 澄清 → 合并」迭代循环逐步完善文档，避免重复处理原始输入浪费 token

---

## 两种运行模式

本 skill 根据输入自动判断运行模式：

### 模式 A：首次生成（Full Run）

**触发条件**：`version-doc/{版本号}/brd/` 中**不存在** `brd_normalized.md`

**输入**：原始 BRD 内容（粘贴文本、文件路径、截图等）

**输出**：
1. **草稿 BRD** `brd_normalized.md`（可能含 `[TBD-xxx]` 占位符）
2. **澄清文件** `clarification_needed.md`（如有待澄清项）

### 模式 B：澄清合并（Incremental Merge）

**触发条件**：`version-doc/{版本号}/brd/` 中**已存在** `brd_normalized.md`，或用户明确提到"补充""合并""更新 BRD""第二轮"

**输入**：
- 已有的草稿 BRD（`brd_normalized.md`）
- 澄清文件（`clarification_needed.md`，业务方已填写回复）
- 可选的额外补充材料

**行为**：
- **不重新处理原始输入**——直接读取草稿 BRD
- 将澄清回复中的内容合并到草稿 BRD 的对应 `[TBD-xxx]` 占位符位置
- 重新检查是否还有新的待澄清项
- 如还有 → 更新草稿 BRD + 生成新的澄清文件
- 如没有 → 移除所有 `[TBD-xxx]` 标记，删除 `clarification_needed.md`，输出最终 BRD

## 占位符规范

当需求中存在不确定、模糊或缺失的信息时，使用结构化占位符标记：

### 占位符格式

```
[TBD-{编号}]: {简要描述缺失的信息}
```

**编号规则**：全局递增，格式为三位数字，如 `TBD-001`、`TBD-002`。

### 占位符使用场景

| 场景 | 占位符示例 |
|------|-----------|
| 业务规则不明确 | `[TBD-001]: 排班调换是否需要审批？` |
| 角色不明确 | `[TBD-002]: "管理员"具体指 HR 还是部门经理？` |
| 数据范围不确定 | `[TBD-003]: 考勤异常的判定标准是什么？` |
| 多处矛盾 | `[TBD-004]: 前面说"仅限正式员工"，后面提到"实习生也适用"，请确认` |
| 优先级存疑 | `[TBD-005]: "假期日历"功能是否为本期必须？` |

### 占位符嵌入方式

占位符直接嵌入到对应需求详情的具体位置，而非集中堆放。例如：

```markdown
**描述**：
员工可以在系统中提交排班调换申请，[TBD-001]: 排班调换是否需要审批？如需审批，审批层级是什么？
```

## 输入

| 参数 | 必填 | 说明 |
|------|------|------|
| **BRD 内容** | ✅ | 直接粘贴文本、指定文件路径、口头描述需求点、或提供截图 |
| **版本号** | ✅ | 如 `v1.0.1`。用于确定输出目录 `version-doc/{版本号}/brd/` |

用户以任意方式提供 BRD 内容：
- 直接粘贴文本（聊天记录、邮件正文、会议纪要等）
- 指定文件路径
- 口头描述需求点
- 提供截图（需先 OCR 提取文字后再处理）
- 草稿 BRD + 澄清文件（二次调用时）

**版本号获取规则**：
1. 如果用户直接指定了版本号 → 使用用户指定的
2. 如果以上未提供 → **向用户询问版本号**，不可跳过

## 输出

**输出位置**：`version-doc/{版本号}/brd/`

| 文件 | 说明 |
|------|------|
| `brd_normalized.md` | 标准化 BRD 文档（草稿或最终版） |
| `clarification_needed.md` | 澄清文件（如有待澄清项；全部解决后删除） |

## 编排流程（模式 A：首次生成）

### 第 0 步：判断运行模式

**0.1 版本号唯一性校验**（🆕 新增）：

检查 `version-doc/{版本号}/` 目录是否已存在：
- 如果存在 → 读取 `version-doc/{版本号}/brd/brd_normalized.md`
  - 如果 BRD 内容与当前需求**明显不同**（标题/背景/需求列表不匹配）→ ⛔ **拒绝覆盖**，输出：
    ```
    ❌ 版本号冲突：version-doc/{版本号}/brd/ 已包含「{已有 BRD 标题}」的 BRD 文档。
    当前需求与已有 BRD 不匹配，请使用新的版本号。
    ```
  - 如果 BRD 内容与当前需求**一致**（同一需求的迭代）→ 进入 **模式 B**（澄清合并）
- 如果不存在 → 进入 **模式 A**（首次生成）

**0.2 判断运行模式**：

检查 `version-doc/{版本号}/brd/` 中是否已有 `brd_normalized.md`：
- 存在 → **模式 B**，跳转到「澄清合并流程」
- 用户明确提到"补充""合并""更新""第二轮" → **模式 B**
- 不存在 → **模式 A**，继续下方流程

### 第 1 步：获取原始内容

- 如果内容过长（超过 3000 字），先做分段标注，确保不遗漏

### 第 2 步：需求拆分

将连续文本拆分为独立的需求点：
- 按段落分隔、序号标记、话题切换等线索拆分
- 粒度标准：一个需求 = 一个独立的功能变更/新增
- 提取每个需求涉及的：角色、功能动作、业务规则、数据字段
- **模糊或缺失的信息用 `[TBD-xxx]` 占位符标记**

### 第 3 步：推断优先级

如果业务方未标注优先级，根据以下规则推断：
- **P0**：使用"必须"、"一定要"、"核心"、"上线前必须"等表述
- **P1**：使用"希望"、"最好"、"需要"等表述
- **P2**：使用"如果可以"、"将来"、"后续"、"锦上添花"等表述
- 所有推断的优先级标注 **（推断）**
- 无法推断的用 `[TBD-xxx]` 标记

### 第 4 步：生成术语表

扫描全文提取业务专有名词、缩写、可能有歧义的术语，给出简明解释。不确定的标注"待确认"并生成对应 `[TBD-xxx]`。

### 第 5 步：生成草稿 BRD

读取 `{SKILL_DIR}/references/output-template.md`，按模板生成 `brd_normalized.md`。

**关键要求**：
- 即使存在待澄清项，也必须生成完整结构的草稿 BRD
- 不确定的地方使用 `[TBD-xxx]` 占位符嵌入到具体位置
- 每个需求都必须包含"原始表述"引用块

**文档状态标记**：
- 如有任何 `[TBD-xxx]` 占位符 → 在文档头部标注 `> ⚠️ 文档状态：**草稿**（含 {N} 个待澄清项，请查看 clarification_needed.md）`
- 如无占位符 → 在文档头部标注 `> ✅ 文档状态：**最终版**`

### 第 6 步：生成澄清文件（如需要）

统计草稿 BRD 中所有 `[TBD-xxx]` 占位符，如果存在，生成 `clarification_needed.md`。

格式见 `{SKILL_DIR}/references/output-template.md` 中的「澄清文件模板」。

**如果没有任何待澄清项**，不生成澄清文件，直接进入第 7 步。

### 第 7 步：输出摘要

**如为草稿（有待澄清项）**：

> 📋 BRD 标准化完成（草稿版），含 {N} 个待澄清项。
> 建议下一步：
> 1. 将 `clarification_needed.md` 发给业务方填写回复
> 2. 业务方填写完成后，再次调用本 skill 即可合并

**如为最终版（无待澄清项）**：

> ✅ BRD 标准化完成，无待澄清事项。
> 建议下一步：使用 `prd-brd-to-prd` skill 将 BRD 转化为 PRD。

## 澄清合并流程（模式 B）

当检测到已有草稿 BRD 时执行。**核心目标：不重新处理原始输入，只读草稿 BRD 做增量合并。**

### B-1 步：读取草稿 BRD

读取 `version-doc/{版本号}/brd/brd_normalized.md`。

**不要重新处理原始输入内容。**

### B-2 步：读取澄清回复

读取业务方已填写的 `clarification_needed.md`：
- 解析每个 `TBD-xxx` 编号和对应的回复内容
- 如果有额外补充材料（如用户又粘贴了新的聊天记录），也需处理

### B-3 步：合并到草稿 BRD

逐一处理每个已回复的 `TBD-xxx`：

1. 在草稿 BRD 中搜索 `[TBD-xxx]` 占位符
2. 用澄清回复的内容替换占位符
3. 如果回复内容需要结构化（如包含多条业务规则），按 BRD 模板格式整理后替换
4. 如果回复中又引出新的疑问，在替换内容中嵌入新的 `[TBD-xxx]` 占位符（编号延续）

### B-4 步：重新检查

合并完成后，扫描整个文档：

1. 统计剩余的 `[TBD-xxx]` 占位符数量
2. 检查新补充的内容是否引入了新的矛盾或不确定性
3. 如果有新发现的问题，生成新的 `[TBD-xxx]` 占位符

### B-5 步：更新输出

**如果仍有 `[TBD-xxx]` 占位符**：
- 更新草稿 BRD（替换已回复的占位符，保留/新增未回复的）
- 更新文档头部的状态标记和待澄清数量
- 重新生成 `clarification_needed.md`（只包含剩余未解决的项）
- 输出摘要，提示业务方继续澄清

**如果所有 `[TBD-xxx]` 占位符都已解决**：
- 移除所有 TBD 相关标记
- 将文档状态从「草稿」更新为「最终版」
- **删除 `clarification_needed.md`**（已无需澄清）
- 输出摘要，提示可以进入 PRD 阶段

## Few-Shot 示例

### 示例 1：首次生成（含待澄清项）

**原始输入**：
> 张总说这次改版必须把首页的搜索功能优化一下，现在用户反馈搜索太慢了，而且搜索结果不准。另外如果可以的话，把搜索历史记录也加上。还有就是商品详情页的分享按钮要能分享到微信和朋友圈。

**草稿 BRD 片段**：
```markdown
> ⚠️ 文档状态：**草稿**（含 2 个待澄清项，请查看 clarification_needed.md）

### 需求 1：搜索性能优化

**描述**：
优化首页搜索响应速度。[TBD-001]: 当前搜索响应时间是多少？目标优化到多少毫秒以内？

**原始表述**：
> 张总说这次改版必须把首页的搜索功能优化一下，现在用户反馈搜索太慢了
```

**澄清文件 `clarification_needed.md` 片段**：
```markdown
### TBD-001
- **位置**：需求 1 > 搜索性能优化 > 描述
- **问题**：当前搜索响应时间是多少？目标优化到多少毫秒以内？
- **上下文**：业务方只说"搜索太慢"，但未给出具体的性能指标
- **建议确认**：① 当前平均响应时间 ② 期望目标 ③ 是否区分简单搜索和复杂搜索
- **回复**：_（请业务方在此填写）_
```

### 示例 2：澄清合并（第二轮）

**业务方回复**：
```markdown
### TBD-001
- **回复**：现在搜索大概 3-5 秒，希望优化到 1 秒以内。不用区分搜索类型，所有搜索都要快。
```

**合并后的 BRD 对应位置**：
```markdown
### 需求 1：搜索性能优化

**描述**：
优化首页搜索响应速度。当前搜索响应时间约 3-5 秒，目标优化到 1 秒以内。所有搜索类型统一要求，不区分简单搜索和复杂搜索。
```

## 约束

### 通用约束
1. **保留原始表述**：每个需求详情中必须包含"原始表述"引用块，便于追溯
2. **不要自行补充需求**：只整理和结构化业务方说的内容，不添加业务方未提及的功能
3. **不做业务质疑**：本 skill 只负责格式转换，不判断需求是否合理
4. **格式严格**：输出必须严格遵循模板，不得遗漏任何章节
5. **占位符可追踪**：每个 `[TBD-xxx]` 占位符必须同时出现在草稿 BRD 的具体位置和澄清文件的问题列表中，两处编号一一对应
6. **模式 B 节约 token**：澄清合并模式下，禁止重新读取/处理原始输入，只操作草稿 BRD 和澄清文件

### 需求拆分反模式（禁止）
- ❌ 把背景描述拆分为独立需求（"目前系统响应慢"是背景，不是需求）
- ❌ 把 UI 样式描述和业务逻辑混在同一需求中
- ❌ 把同一功能的正向描述和反向描述拆为两个需求（"支持导出 + 导出格式为 Excel"是一个需求）
- ❌ 拆分粒度过细（"按钮颜色改为蓝色"不应独立成需求，应归入所属功能需求）

### 边界条件处理
- 内容互相矛盾 → 两处都保留，标注"⚠️ 矛盾，待澄清"，生成 `[TBD-xxx]`
- 内容过于模糊无法拆分 → 保留原文作为一个需求，嵌入 `[TBD-xxx]` 请求补充
- 原始内容只有一句话 → 仍生成完整结构，缺失部分用 `[TBD-xxx]` 标记
- 术语表为空 → 保留空表，注明"未发现专有术语"
- 澄清回复不完整 → 已回复的照常合并，未回复的保持 `[TBD-xxx]` 原样
- 澄清回复引入新问题 → 在替换位置嵌入新的 `[TBD-xxx]`，编号延续
