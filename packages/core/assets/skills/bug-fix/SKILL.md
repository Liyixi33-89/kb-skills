# Bug-Fix — Bug 修复

## 目标

基于用户提供的错误描述、错误日志或复现步骤，系统化地定位问题根因，生成修复代码，并验证修复不引入新问题。

**核心隐喻**：你是一个"资深排障工程师"——不急于修改代码，先理解问题全貌。定位根因比修复症状更重要。每次修复都要确认影响范围。

---

## 设计原则

1. **根因优先**：先定位根因，再修复。不做"头痛医头"的表面修复
2. **KB 加速定位**：利用知识库快速定位相关模块，避免在整个代码库中盲目搜索
3. **最小修改**：修复代码的改动范围尽可能小，不趁机重构
4. **影响分析**：修复前分析影响范围，确保不破坏其他功能
5. **可验证**：每个修复都附带验证步骤，确认问题已解决

---

## 输入

| 参数 | 必填 | 说明 |
|------|------|------|
| **问题描述** | ✅ | 错误描述、错误日志、复现步骤（至少提供一项） |
| **错误日志** | 否 | 控制台报错、服务端日志、网络请求错误等 |
| **复现步骤** | 否 | 触发 bug 的操作步骤 |
| **期望行为** | 否 | 正确的行为应该是什么 |
| **影响范围** | 否 | 哪些功能/页面受影响 |

## 输出

| 输出 | 说明 |
|------|------|
| 根因分析报告 | 问题定位过程 + 根因说明 |
| 修复代码 | 直接修改源码文件 |
| 验证步骤 | 如何确认 bug 已修复 |
| 回归检查 | 需要额外检查的关联功能 |

---

## 编排流程

### 第 1 步：理解问题

分析用户提供的信息，提取关键线索：

| 线索类型 | 提取内容 |
|---------|---------|
| 错误信息 | 错误类型、错误消息、堆栈跟踪 |
| 触发条件 | 什么操作触发、什么数据触发、是否可稳定复现 |
| 影响范围 | 哪些页面/功能受影响 |
| 时间线 | 什么时候开始出现（是否与某次变更相关） |

**如果信息不足**：
- 列出需要补充的信息
- 但不要因此停止——先基于已有信息进行初步分析

### 第 2 步：KB 快速定位

根据错误线索，从 KB 中定位相关模块：

| 线索 | 查找策略 |
|------|---------|
| API 路径（如 `/api/agents`） | `kb/server/server/01_index_api.md` → 找到对应路由文件 |
| 页面名称（如"Agent 列表页"） | `kb/frontend/.../01_index_page.md` → 找到对应页面文件 |
| Model 名称（如"Agent"） | `kb/server/server/02_index_model.md` → 找到 Schema 定义 |
| 错误消息关键词 | 在 KB 索引中搜索，定位可能的模块 |
| 组件名称 | `kb/frontend/.../02_index_component.md` → 找到组件定义 |

**输出**：初步定位的文件列表（可能涉及的 2-5 个文件）

### 第 3 步：源码深入分析

读取第 2 步定位的源码文件，进行深入分析：

#### 3.1 调用链追踪

从错误发生点开始，追踪完整的调用链：

```
前端页面 → API 调用函数 → HTTP 请求 → 后端路由 → Service → Model → MongoDB
```

在调用链的每个节点检查：
- 参数传递是否正确
- 返回值处理是否正确
- 错误处理是否完整

#### 3.2 根因定位

常见根因类型：

| 根因类型 | 特征 | 检查方法 |
|---------|------|---------|
| 类型错误 | `TypeError`、`undefined is not a function` | 检查变量类型和 null 检查 |
| 逻辑错误 | 功能行为不符合预期 | 对照设计文档检查业务逻辑 |
| 数据问题 | 数据库中数据不符合 Schema | 检查数据校验和迁移 |
| 并发问题 | 偶发性错误、竞态条件 | 检查异步操作和状态管理 |
| 配置问题 | 环境变量缺失、配置错误 | 检查 .env 和配置文件 |
| 依赖问题 | 版本不兼容、API 变更 | 检查 package.json 和 lock 文件 |

### 第 4 步：影响范围分析

在修复前，分析修复可能影响的范围：

1. **直接影响**：修改的文件本身
2. **上游影响**：调用修改文件的其他文件
3. **下游影响**：修改文件调用的其他文件
4. **数据影响**：是否需要修复已有的脏数据

```markdown
### 影响范围

| 文件 | 影响类型 | 说明 |
|------|---------|------|
| server/src/routes/agents.ts | 直接修改 | 修复参数校验 |
| server/src/services/agentService.ts | 间接影响 | 需确认调用方式不变 |
| web/src/pages/AgentsPage.tsx | 间接影响 | 需确认前端适配 |
```

### 第 5 步：生成修复代码

**修复原则**：
1. **最小改动**：只修改必要的代码，不趁机重构
2. **添加注释**：在修复处添加注释说明为什么这样改
3. **保持风格**：修复代码的风格与周围代码一致

**修复代码格式**：

```markdown
### 修复文件：{filePath}

**修复前**：
```typescript
// L{lineNumber}
{buggyCode}
```

**修复后**：
```typescript
// L{lineNumber}
{fixedCode}  // fix: {简要说明}
```

**修复说明**：{为什么这样修复}
```

### 第 6 步：验证方案

```markdown
### 验证步骤

1. **复现验证**：按原始复现步骤操作，确认 bug 不再出现
2. **正向验证**：{正常功能仍然正常的验证步骤}
3. **边界验证**：{边界条件的验证步骤}

### 回归检查清单

- [ ] {关联功能 1} 仍正常工作
- [ ] {关联功能 2} 仍正常工作
- [ ] {关联功能 3} 仍正常工作
```

### 第 7 步：输出摘要

```
## Bug 修复完成

### 问题摘要
{一句话描述问题}

### 根因
{一句话描述根因}

### 修复文件
| # | 文件 | 修改内容 |
|---|------|---------|
| 1 | {file} | {change} |

### 验证
- 复现验证：{通过/待验证}
- 回归检查：{N} 个关联功能需确认

建议下一步：
1. 按验证步骤确认修复有效
2. 使用 `code-review` 审查修复代码
3. 使用 `gen-test-code` 为该场景补充测试用例（防止回归）
4. 执行 `doc-code-to-kb` 更新知识库
```

## 约束

### 修复约束

1. **不做无关修改**：只修复 bug，不趁机重构或优化
2. **不删除功能**：修复不能导致其他功能失效
3. **保留错误处理**：不能通过"吞掉错误"来"修复"问题
4. **标注不确定性**：如果根因不完全确定，标注 `⚠️ 推测根因`

### 边界条件处理

| 场景 | 处理方式 |
|------|---------|
| 信息不足以定位 | 列出需要补充的信息，同时给出最可能的方向 |
| KB 不存在 | 直接搜索源码，标注"⚠️ 未参考 KB" |
| 问题涉及第三方库 | 检查版本兼容性，建议升级或降级 |
| 问题无法在本地复现 | 分析可能的环境差异，给出排查方向 |
| 修复需要数据迁移 | 标注需要配合 `db-migration` Skill |
| 根因在多个文件中 | 按依赖顺序逐一修复 |
