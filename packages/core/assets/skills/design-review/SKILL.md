# Design-Review — 设计评审

## 目标

在 Pipeline 的规划阶段（Phase 1）完成后，自动汇总所有设计文档的变更内容，生成结构化的**变更清单摘要**，标注**关键设计决策点**，输出**影响范围分析**，为人工审批提供清晰、完整的决策依据。

**核心隐喻**：你是一个"架构评审委员会的秘书"——不做决策，但把所有需要决策的信息整理得清清楚楚，让决策者一目了然。

**核心理念（Review-First）**：借鉴 Kiro 的 Spec-Driven Development——设计文档是编码的"合同"，人工审批后才能进入编码阶段。AI 不是起点，而是终点；设计评审才是编码的起点。

---

## 设计原则

1. **信息完整**：变更清单必须覆盖所有设计文档中定义的新增/修改文件，不遗漏
2. **决策显式化**：所有需要人工判断的设计决策都必须标注 `⚠️`，不能隐藏在文档深处
3. **影响可追溯**：每个变更都标注影响范围（影响哪些现有文件、哪些现有功能）
4. **不做决策**：只呈现信息和选项，不替人工做设计决策
5. **格式固定**：输出格式固定，支持跨版本对比和团队协作

---

## 输入

| 参数 | 必填 | 说明 |
|------|------|------|
| **版本号** | ✅ | 如 `v1.4.0`，用于定位设计文档目录 |
| **设计文档目录** | 否 | 默认 `version-doc/{版本号}/design/` |
| **PRD 文件** | 否 | 默认 `version-doc/{版本号}/prd/prd.md`，用于交叉验证 |
| **Story 文件** | 否 | 默认 `version-doc/{版本号}/stories/`，用于关联 Story |

## 输出

| 文件 | 说明 |
|------|------|
| `version-doc/{版本号}/review/design-review.md` | 设计评审摘要文件（持久化） |
| 控制台输出 | 评审摘要 + 人工审批提示 |

## 编排流程

### 第 0 步：检查前置条件

1. 检查 `version-doc/{版本号}/design/` 目录是否存在
   - 不存在 → 提示用户先执行规划阶段（Phase 1）的 Skill
2. 检查设计文档是否完整：
   - `be-*.md`（后端设计）是否存在
   - `fe-*.md`（前端设计）是否存在
   - `ui-spec.md`（UI 设计规范）是否存在
   - 缺失的标注 `⚠️ 缺失`，但不阻塞评审

### 第 1 步：读取所有设计文档

读取 `version-doc/{版本号}/design/` 下的所有设计文档，提取：

**从后端设计文档（`be-*.md`）提取**：
- 新增/修改的 Model 列表（字段定义、索引）
- 新增/修改的 API 端点列表（路径、方法、参数、响应）
- 新增/修改的 Service 函数列表
- 数据库变更（新增 Collection、新增索引、修改 Schema）
- 中间件/配置变更

**从前端设计文档（`fe-*.md`）提取**：
- 新增/修改的页面列表（路由、功能）
- 新增/修改的组件列表（Props 接口）
- 新增/修改的 API 封装函数
- Store 变更
- 类型定义变更
- 路由变更

**从 UI 设计规范（`ui-spec.md`）提取**：
- 页面布局和交互流程
- 组件使用规范

### 第 2 步：读取 KB 分析影响范围

读取项目知识库，对比设计文档中的变更：

| 变更类型 | KB 对比 | 影响分析 |
|---------|---------|---------|
| 修改现有 Model | 读取 `kb/server/server/02_index_model.md` | 哪些 Service/Route 引用了该 Model |
| 修改现有 API | 读取 `kb/server/server/01_index_api.md` | 哪些前端页面调用了该 API |
| 修改现有页面 | 读取 `kb/frontend/*/01_index_page.md` | 页面的现有功能是否受影响 |
| 修改共享文件 | 读取 `api/index.ts`、`types/index.ts` | 是否与其他功能冲突 |

### 第 3 步：交叉验证 PRD 覆盖度

如果 PRD 文件存在，验证设计文档是否覆盖了 PRD 中的所有功能需求：

```markdown
### PRD 功能覆盖度

| PRD 功能模块 | 后端设计 | 前端设计 | 覆盖状态 |
|-------------|---------|---------|---------|
| 功能 1: xxx | be-*.md §3.1 | fe-*.md §2.1 | ✅ 已覆盖 |
| 功能 2: xxx | be-*.md §3.2 | — | ⚠️ 前端未覆盖 |
| 功能 3: xxx | — | — | ❌ 未覆盖 |
```

### 第 4 步：识别关键设计决策点

扫描设计文档，识别需要人工确认的设计决策：

**自动识别规则**：
1. **数据模型选择**：字段类型、关联方式（引用 vs 嵌入）、索引策略
2. **API 设计**：RESTful 路径设计、请求/响应格式、分页策略
3. **权限模型**：数据归属（用户级 vs 全局级）、访问控制策略
4. **级联操作**：删除时的关联数据处理（级联删除 vs 软删除 vs 解除引用）
5. **性能策略**：是否需要缓存、是否需要异步处理
6. **技术选型**：是否引入新的依赖库

**决策点格式**：
```markdown
### ⚠️ 关键设计决策

| # | 决策点 | 当前方案 | 备选方案 | 影响 |
|---|--------|---------|---------|------|
| D-001 | Tag 数据归属 | 用户级（每个用户独立标签） | 全局级（所有用户共享） | 影响 Model 设计和查询逻辑 |
| D-002 | Agent.tags 存储方式 | 存 tagId（引用关系） | 存 tagName（冗余存储） | 影响查询性能和数据一致性 |
| D-003 | 删除标签时的处理 | 级联移除 Agent 中的引用 | 软删除标签 | 影响数据完整性 |
```

### 第 5 步：生成变更清单摘要

汇总所有信息，生成结构化的评审摘要：

- 前端设计：`version-doc/{v}/design/fe-{feature}.md`
- UI 规范：`version-doc/{v}/design/ui-spec.md`

## 约束

### 评审约束

1. **不做设计决策**：只呈现信息和选项，不替人工做决策
2. **不修改设计文档**：评审只输出摘要，不直接修改设计文档（除非人工要求"修改后确认"）
3. **变更清单必须完整**：不遗漏任何设计文档中定义的新增/修改文件
4. **影响范围必须准确**：基于 KB 中的实际代码引用关系分析，不凭空猜测
5. **决策点必须显式化**：所有需要人工判断的设计决策都必须标注，不能隐藏

### `--auto` 模式约束

1. `--auto` 模式下仍然执行完整的评审流程，生成评审摘要文件
2. 评审摘要中标注 `⚡ 自动通过（--auto 模式）`
3. 关键设计决策点仍然标注 `⚠️ 待确认`（供后续追溯）
4. `--auto` 模式不影响 Gate 2（代码审查），Gate 2 始终执行

### 边界条件处理

| 场景 | 处理方式 |
|------|---------|
| 设计文档不存在 | 提示用户先执行规划阶段 |
| 只有后端设计，无前端设计 | 只评审后端部分，标注"⚠️ 前端设计缺失" |
| 只有前端设计，无后端设计 | 只评审前端部分，标注"⚠️ 后端设计缺失" |
| KB 不存在 | 跳过影响范围分析，标注"⚠️ 无 KB，未分析影响范围" |
| PRD 不存在 | 跳过 PRD 覆盖度验证，标注"⚠️ 无 PRD，未验证功能覆盖度" |
| 设计文档过于简略 | 标注"⚠️ 设计文档信息不足，建议补充后再评审" |
| 修改后确认超过 3 轮 | 暂停评审，建议重新执行规划阶段 |
