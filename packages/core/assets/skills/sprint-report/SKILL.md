# Sprint-Report — Sprint 回顾报告

## 目标

在 Sprint 结束时，自动收集本迭代的所有开发数据（Story 完成情况、Quality Gate 记录、代码变更统计、Pipeline 执行记录），生成结构化的 Sprint 回顾报告，为团队提供数据驱动的回顾依据。

**核心隐喻**：你是一个"数据分析师 + Scrum Master"——用数据说话，不凭感觉。每个结论都有数据支撑，每个改进建议都有具体的行动项。

---

## 设计原则

1. **数据驱动**：所有指标从实际文件中提取，不凭空估算
2. **双视角**：同时面向管理层（业务维度）和开发团队（技术维度）
3. **可对比**：报告格式固定，支持跨 Sprint 对比
4. **可行动**：改进建议必须具体、可执行，不说空话
5. **自动化**：尽可能从已有文件中提取数据，减少手动输入

---

## 输入

| 参数 | 必填 | 说明 |
|------|------|------|
| **版本号** | ✅ | 本 Sprint 对应的版本号 |
| **Sprint 编号** | 否 | 如 `Sprint 5`，默认自动递增 |
| **Sprint 周期** | 否 | 如 `2024-01-15 ~ 2024-01-28`，默认 2 周 |
| **团队规模** | 否 | 默认 1 人（AI 辅助开发） |

## 编排流程

### 第 1 步：收集 Story 完成数据

读取 `version-doc/{版本号}/stories/` 下的所有 Story 文件：

对每个 Story，检查其验收标准的完成情况：
- 所有验收标准都已勾选 `[x]` → ✅ 完成
- 部分验收标准已勾选 → 🟡 部分完成（计算完成百分比）
- 无验收标准被勾选 → ❌ 未完成

```markdown
### Story 完成统计

| Story | 标题 | 优先级 | 估算工时 | 实际工时 | 状态 | 完成度 |
|-------|------|--------|---------|---------|------|--------|
| 1.1 | Agent Model 扩展 | P0 | 0.5d | 0.5d | ✅ | 100% |
| 1.2 | 标签 API | P0 | 1d | 1.5d | ✅ | 100% |
| 1.3 | 标签筛选 UI | P0 | 1.5d | — | ❌ | 0% |
| 2.1 | TagSelector 组件 | P1 | 1d | 0.5d | ✅ | 100% |
```

### 第 4 步：计算关键指标

#### 4.1 完成率

```
Story 完成率 = 已完成 Story 数 / 总 Story 数 × 100%
验收标准完成率 = 已通过验收条件数 / 总验收条件数 × 100%
计划工时完成率 = 实际完成工时 / 计划总工时 × 100%
```

#### 4.2 质量指标

```
首次 Review 通过率 = 首次通过的文件数 / 总审查文件数 × 100%
缺陷密度 = 🔴 问题数 / 新增代码千行数
测试覆盖率 = 已覆盖验收标准数 / 总验收标准数 × 100%
```

#### 4.3 效率指标

```
平均 Story 完成时间 = 总实际工时 / 已完成 Story 数
工时偏差率 = (实际总工时 - 计划总工时) / 计划总工时 × 100%
Pipeline 平均执行时间 = 总 Pipeline 时间 / Pipeline 执行次数
```

### 第 5 步：分析风险和改进点

#### 5.1 风险分析

| 风险类型 | 检测方法 | 示例 |
|---------|---------|------|
| 进度风险 | Story 完成率 < 80% | "本 Sprint 完成率仅 60%，3 个 Story 未完成" |
| 质量风险 | 首次 Review 通过率 < 50% | "代码质量需要提升，50% 的代码首次审查未通过" |
| 技术债务 | 🟡 问题累积 > 10 个 | "累积 15 个建议修复项未处理" |
| 估算偏差 | 工时偏差率 > 30% | "实际工时超出计划 40%，估算需要校准" |

#### 5.2 改进建议

每个改进建议必须包含：
- **问题**：具体的问题描述
- **数据**：支撑的数据指标
- **行动**：具体的改进行动
- **负责**：谁来执行
- **时间**：什么时候完成

## 约束

### 报告约束

1. **数据必须真实**：所有指标从实际文件中提取，不估算
2. **趋势需要历史数据**：如果是第一个 Sprint，趋势列标注"—"
3. **改进建议必须可行**：每个建议都有具体的行动项和时间
4. **不做主观评价**：不说"团队表现优秀"这种主观评价，用数据说话

### 边界条件处理

| 场景 | 处理方式 |
|------|---------|
| Story 文件不存在 | 标注"⚠️ 无 Story 文件"，跳过完成率分析 |
| Pipeline 状态文件不存在 | 标注"⚠️ 无 Pipeline 记录"，跳过 Quality Gate 分析 |
| 无 Code Review 记录 | 标注"⚠️ 未执行代码审查" |
| 无测试文件 | 标注"🔴 无测试覆盖" |
| 第一个 Sprint | 趋势列标注"—"，不做跨 Sprint 对比 |
| 所有 Story 都完成 | 仍然分析质量指标和改进空间 |
